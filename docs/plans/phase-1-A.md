**Dependencies:** none — this workstream is the unblocker for the rest of phase 1.

## Task 1 — `requires_role(*roles)` dependency

**Files:** `backend/app/deps.py`, `backend/tests/test_rbac.py`.

The MVP `get_current_org` already loads the first membership; extend with a factory:

```python
def requires_role(*roles: str):
    async def _dep(user: CurrentUser, session: DbSession) -> Membership:
        result = await session.execute(
            select(Membership).where(Membership.user_id == user.id).order_by(Membership.created_at.asc()).limit(1)
        )
        m = result.scalar_one_or_none()
        if m is None or m.role not in roles:
            raise HTTPException(403, detail={"code": "forbidden", "message": "Requires role"})
        return m
    return Depends(_dep)
```

Tests must cover:
- owner ✓ for `requires_role("owner")`.
- admin ✓ for `requires_role("owner", "admin")`.
- member ✗ → 403.
- no membership → 403.

Commit: `feat(phase-1.A): requires_role dependency`.

## Task 2 — `Invitation` + `PasswordResetToken` models and migration

**Files:** new models, `__init__`, `alembic/env.py`, alembic migration.

Invitation:
- `id` UUID PK
- `org_id` FK to orgs, indexed
- `email` String(255), indexed (case-insensitive lookup)
- `role` String(32) — `"admin"` | `"member"`
- `token_hash` String(64) — sha256 hex of the raw token; never store raw
- `created_by_user_id` FK to users
- `expires_at` DateTime tz, default 7 days from now (set by service)
- `accepted_at` DateTime tz nullable
- `revoked_at` DateTime tz nullable
- `created_at` DateTime tz default `now()`

PasswordResetToken:
- `id` UUID PK
- `user_id` FK to users, indexed
- `token_hash` String(64)
- `expires_at` DateTime tz, default 1 hour from creation
- `consumed_at` DateTime tz nullable
- `created_at` DateTime tz default `now()`

Migration via `alembic revision -m "..." --autogenerate`. Run `alembic upgrade head` to verify clean apply. Confirm autogenerate captured both indexes.

Commit: `feat(phase-1.A): invitations + password_resets tables`.

## Task 3 — Token service

**Files:** `backend/app/services/auth/tokens.py`, `backend/app/services/auth/__init__.py`, tests inline within test_invitations.

```python
import hashlib, secrets

def generate_token() -> tuple[str, str]:
    raw = secrets.token_urlsafe(32)
    return raw, hashlib.sha256(raw.encode()).hexdigest()

def hash_token(raw: str) -> str:
    return hashlib.sha256(raw.encode()).hexdigest()
```

Plus `verify_invitation_token` / `consume_invitation_token` that take a session + raw token, look up by hash, check status + expiry, mark `accepted_at`. Same shape for reset tokens.

Commit: `feat(phase-1.A): token generation + verification service`.

## Task 4 — Password reset endpoints

`POST /auth/forgot-password` (no auth):
- Body: `{ email }`.
- Always return `{ ok: true }` (don't leak whether the email exists).
- If user exists, create a `PasswordResetToken`. In dev (`settings.env != "prod"`), include the raw token in the response under `dev_token`. Otherwise omit.
- Audit: `record_audit(action="auth.password_reset_requested", entity_type="user", entity_id=user.id, ...)`. Skip audit if user doesn't exist (no-op).

`POST /auth/reset-password`:
- Body: `{ token, new_password }`.
- Verify and consume the token. Update `users.password_hash`. Revoke any other unconsumed tokens for the same user.
- Audit: `action="auth.password_reset_consumed"`.
- On failure (expired/consumed/unknown), return 400 with `{code: "invalid_token"}` — no user-existence leak.

Tests: happy path, expired token, double-use rejected, wrong token rejected.

Commit: `feat(phase-1.A): password reset round-trip`.

## Task 5 — Invitations endpoints (admin-only)

All routes under `/org/invitations`, gated `requires_role("owner", "admin")` except the accept path which is on `/auth/accept-invite`.

- `POST /org/invitations` body `{ email, role }` → creates invitation, returns `InvitationDTO` with `dev_token` in non-prod envs.
- `GET /org/invitations` → list pending invites for the current org.
- `DELETE /org/invitations/{id}` → revoke (sets `revoked_at`).
- `POST /auth/accept-invite` body `{ token, email, password? }`:
  - Verify token. If `email` matches an existing user, attach a Membership with the invited role.
  - If no user, create the user (`password` required), then attach Membership.
  - Mark invitation `accepted_at`.
  - Return a fresh `SessionDTO` (auto-login the new member).

Tests: full round-trip; double-accept rejected; expired rejected; role gate keeps members out of `POST /org/invitations`; audit log entries on every mutation.

Commit: `feat(phase-1.A): magic-link invitations + accept`.

## Hard rules

- TDD per task. Failing test → make pass → refactor.
- Every mutation route calls `record_audit(...)`.
- Every mutation route publishes the corresponding event (`membership.role_changed`, `invitation.created`, etc.) for SSE clients via `publish_event(...)` from 0.B. Use try/except so a Redis failure can't roll back the DB write.
- `requires_role` is server-side enforcement; frontend `lib/permissions.ts` is UX only.
- No raw tokens ever logged or returned in prod env.
- No skipping the dev playground — manually exercise the team page once after wiring.
- No new dependencies needed.
