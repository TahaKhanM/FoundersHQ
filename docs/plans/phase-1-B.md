# Phase 1.B — Onboarding Wizard

> **For agentic workers:** REQUIRED SUB-SKILL: `superpowers:subagent-driven-development` or `superpowers:executing-plans`. Steps use `- [ ]`.

**Goal:** a first-run user goes through a multi-step onboarding wizard that captures org name, base currency, fiscal year, persona, and either seeds sample data, imports a CSV, or starts empty. Progress persists so users can resume.

**Architecture:** wizard lives in a dedicated route group `app/(onboarding)/onboarding/` outside `(shell)` (no nav rail). Step state lives in URL query params (`?step=2`) plus persisted backend rows so multi-device works. Sample-data path invokes the existing `seed_dev_data` script scoped to the user's new org via a new endpoint.

**Tech Stack:** Next.js 16 App Router (route groups), react-hook-form + zod, Money component, existing FinancialProfile model.

**Dependencies:** 1.A merged (so we have the role gate + audit helper). The wizard itself is owner-only.

**Skills:** `foundershq-conventions`, `superpowers:test-driven-development`, `vercel:nextjs`, `frontend-design:frontend-design`.

## Tasks (compressed)

1. **Backend model + migration** — add the six org columns. Test: `Org(base_currency="EUR").base_currency == "EUR"`.
2. **State machine service** — pure function `next_step(current, payload) -> NextStep`. Unit tests cover: invalid payload → ValidationError; valid step 1 payload → "persona"; step 3 with sample-data option → "seed"; etc.
3. **Onboarding router** — `GET /onboarding/state` returns `{ step: int, completed_at: datetime | None, captured: {...} }`. `POST /onboarding/step/{n}` validates payload, persists, returns next state. `POST /onboarding/complete` requires all steps captured; sets `onboarding_completed_at`. Every mutation audits.
4. **Seed-sample-data endpoint** — owner-only; refactors `scripts/seed_dev_data.py` to take an `org_id` param. Imports a small fixture (10 transactions, 3 invoices, 2 commitments) so the dashboard isn't empty.
5. **Frontend wizard shell** — `(onboarding)/layout.tsx`, the wizard page reading `step` from URL, progress bar.
6. **Step 1 (org)** — react-hook-form + zod; org name + currency (drop-down with top 8) + fiscal year start month. Persist on submit.
7. **Step 2 (persona)** — radio group: "founder operator", "first-time founder", "second-time founder", "ops/finance lead". Maps to persona string.
8. **Step 3 (data)** — three cards: "Seed sample data" (calls seed endpoint), "Import CSV" (deep-link to existing CSV ingest at `/spending/transactions`), "Start empty" (no-op, advances).
9. **Step 4 (done)** — congratulatory message; CTA "Go to Dashboard".
10. **Shell middleware redirect** — if user is logged in, `onboarding_completed_at` is null, AND the org has no rows, redirect any `(shell)` route to `/onboarding`.

## Tests

- `test_onboarding_state.py` — every transition.
- `test_onboarding_router.py` — full happy path via the test client.
- Playwright: cold-start sign-up → onboarding completes in 4 steps → dashboard renders sample data.

## Hard rules

- No new dependencies.
- Wizard step payloads are typed (Pydantic discriminated union).
- All mutations write audit entries.
- Persona enum lives in `app/models/org.py` next to `OrgRole`.
- No business logic in steps' tsx files; hooks call the API.

## Definition of done

- `cd backend && make verify` clean; new tests pass.
- `cd frontend && pnpm verify` clean.
- Fresh sign-up → onboarding → seeded dashboard works in `pnpm dev` (Playwright snapshot).
- A user who completed onboarding cannot reach `/onboarding` again (redirected to `/dashboard`).
